¿Qué significa "gobernar" procesos?

  • Un proceso es un programa en ejecución: su estado, su memoria, sus descriptores y un estado de planificación en el que el SO decide cuándo le da CPU.
  • El SO ya gobierna procesos (planificador, memoria…) pero de forma genérica y justa para todos: no conoce las políticas de mi organización.
  • top/htop solo dejan observar: ante un proceso desbocado, el admin cierra el lazo a mano.

De observación pasiva a gobernanza activa

ProcGuard cierra el lazo de control automáticamente, gobernado por políticas declarativas (archivo INI):

Observación→ Detección→ Decisión→ Acción→ Registro↻

Es un sistema de control realimentado sobre los procesos: observa, decide, actúa… y vuelve a observar.

Pilar 1 · Observar

La frontera kernel / espacio de usuario

  • Un programa de usuario no puede leer la memoria del kernel: la separación por anillos de protección lo impide.
  • El SO expone su estado vía procfs, un pseudo-sistema de archivos: los archivos de /proc no están en disco — el kernel los genera al vuelo al leerlos. Es la ventana oficial del kernel.
  • De ahí leo el estado del proceso (ejecutándose, durmiendo, detenido, zombi) y el árbol de procesos (PPID); los huérfanos los readopta init (PID 1).

Pilar 1 · Observar

Identidad de un proceso y reciclaje de PID

  • El PID no es una identidad permanente: es un recurso finito que el SO recicla cuando un proceso muere.
  • Si decido "matar al PID 1234" y ese número fue reasignado entre la observación y la acción, mataría al proceso equivocado.
  • Solución: identifico por la tupla (PID, tiempo de inicio). El tiempo de inicio lo fija el kernel al crear el proceso y no se repite.

Pilar 2 · Medir

La contabilidad de recursos del kernel

%CPU = una tasa

El kernel solo da tiempos acumulados (ticks de reloj). El % es Δcpu / Δtiempo entre dos muestras.

Reloj monótono

Para el "tiempo real" uso un reloj que no salta con ajustes de hora (si no, tasas negativas).

Memoria: física vs virtual

RSS = páginas en RAM; virtual = espacio que el proceso cree tener. La diferencia es la paginación.

Idea clave: el SO mide acumulados; las métricas accionables las deriva el observador.

Pilar 3 · Controlar — decidir

Decidir sin reaccionar al ruido

Un controlador que reacciona a cada pico es inestable. Tres mecanismos contra falsos positivos:

Persistencia

Solo se activa si el umbral se supera durante N muestras seguidas.

Histéresis

Para desactivar, la métrica baja de un 2º umbral y se mantiene (evita el parpadeo).

Cooldown

Enfriamiento entre acciones antes de escalar.

La misma filosofía con la que el SO no expulsa un proceso por un microsegundo de actividad.

Pilar 3 · Controlar — actuar

Las palancas de control del kernel

renice → planificador

Cambia la prioridad (nice): el scheduler le da menos CPU, sin matarlo.

cage → cgroups v2

Confina recursos (límite de CPU) en un grupo de control. Es la base de los contenedores.

stop / kill → señales

Comunicación asíncrona del SO. SIGSTOP suspende, SIGKILL termina — el proceso no puede ignorarlas.

Escalamiento por severidad creciente:

warn→renice→ cage→stop→kill

Pilar 3 · Controlar — con cuidado

Condición de carrera y protección del sistema

  • TOCTOU: entre que observo y actúo hay una ventana; el proceso pudo morir y el PID reciclarse. Por eso revalido (PID, tiempo de inicio) justo antes de cada acción destructiva.

Defensa en profundidad (protege la acción, no la detección):

Dry-run por defecto

No actúa hasta que el admin lo active.

Lista blanca

Nunca toca PID 1, sí-mismo ni servicios críticos.

Cordura 5s + techo

No mata por un pico; límite de acciones.

Mínimo privilegio: para confinar solo escribo en mi propio subárbol de cgroups, nunca en los del sistema.

Concepto · Concurrencia

Tres hilos, mutex y deadlock

  • Tres naturalezas temporales distintas → tres hilos: gobernanza periódica, eventos asíncronos, interfaz reactiva.
  • Comparten datos con mutex, patrón productor–consumidor (colas): un hilo produce, otro consume.
  • Deadlock imposible por diseño: ningún hilo toma más de un mutex a la vez → rompo la espera circular, una de las cuatro condiciones de Coffman.

Las condiciones de carrera no se ven a simple vista → se validan con herramientas dedicadas (ThreadSanitizer). Modelo diseñado; hoy el ciclo corre secuencial.

Concepto · Robustez

Resiliencia ante fallos parciales

  • Observar miles de procesos exige tolerar fallos: al leer un proceso que justo desapareció, el SO devuelve un error (no existe el proceso, permiso denegado).
  • Trato esos como condiciones normales: descarto ese proceso y sigo. El recolector nunca aborta por un proceso individual.
  • Best-effort: si una etapa del ciclo falla, se salta y se registra; el ciclo siguiente arranca limpio. Un fallo parcial no tumba el sistema.
